# Shop! 小组件信息架构、筛选与后台同步设计

日期：2026-07-16

## 目标

重构 iPhone 与 Mac 小组件，使不同尺寸都能有效利用空间，并补齐标签展示、最近完成恢复、快速添加、每实例标签筛选和低频后台同步。

本次设计优先保证：

- 小组件上的完成与恢复操作即时反馈。
- 小组件信息密度随尺寸增加，而不是用时间或装饰性统计填空。
- 每个小组件实例可独立保存标签筛选条件。
- 空闲时低频同步；有小组件操作时合并为一次尽快同步请求。
- 明确遵守系统后台调度限制，不承诺准点或秒级联网同步。

## 非目标

- 不在小组件中直接输入物品名称。WidgetKit 不提供键盘文本输入能力。
- 不在小组件内编辑已有物品的名称、日期或标签。
- 不提供小组件内的刷新按钮。
- 不保证后台任务严格按整点、每小时或每三小时运行。
- 不新增同步服务器、推送服务或账号系统。

## 方案选择

评估过三种标签筛选方式：

1. 小组件内点击标签切换：操作直接，但公开 API 无法可靠识别具体小组件实例，因此不能满足每实例独立持久化。
2. 长按后通过“编辑小组件”配置：由 WidgetKit 保存每个实例的配置，稳定支持不同小组件使用不同筛选条件。
3. 混合模式：编辑配置默认值，小组件内临时切换；状态来源和持久化语义容易混乱。

采用方案 2。快速添加采用深链接打开 App 添加页，因为这是同时支持名称输入、选择已有标签和新建标签的可靠方式。

## 1. 数据快照

扩展 `WidgetSnapshotStore.Snapshot`，使其包含：

- 待买物品 `activeItems`
- 最近完成物品 `recentlyCompletedItems`
- 当前可用的全部标签 `availableTags`
- 快照更新时间 `updatedAt`

物品快照字段：

- `id`
- `name`
- `sortOrder`
- `tags`
- `completedAt`（待买物品为 `nil`）

标签快照字段：

- `id`
- `name`
- `colorHex`

发布规则：

- 待买物品沿用主 App 当前排序。
- 最近完成按 `completedAt` 倒序发布，快照只保留足够小组件使用的有限条数，建议上限 20 条。
- 标签按主 App 现有规则，即名称本地化升序发布。
- 所有新增字段采用向后兼容解码；旧快照缺字段时使用空数组或 `nil`。
- App 启动、数据变更、同步完成后继续统一发布快照并重载相关 timeline。

## 2. 不同尺寸的布局

### 2×2（systemSmall）

- 不显示“Shop!”应用名称。
- 顶部使用紧凑栏：左侧显示当前结果的待买数量，右侧为 `+`。
- 中部优先显示约 2 条待买物品。
- 每条展示完成圆圈、物品名称和标签：
  - 第 1 个标签显示色点与名称。
  - 其余标签只显示颜色圆点。
- 底部用分隔线隔开 1 条最近完成；显示完成图标与名称，点击可恢复为待买。
- 若没有最近完成，不保留空白占位，待买列表可使用剩余高度。

### 2×4（systemMedium）

- 顶部显示“Shop!”、当前结果的待买数量和 `+`。
- 中部优先显示约 3–4 条待买物品。
- 每条右侧最多 3 个标签显示色点与名称，超出部分只显示颜色圆点。
- 底部显示 1 条最近完成，点击可恢复。
- 最近完成紧跟待买列表，不使用把内容强行推到边缘的空白 `Spacer`。

### 4×4（systemLarge）

- 顶部与 2×4 一致。
- 中部使用更多高度显示约 6–8 条待买物品。
- 标签规则与 2×4 一致。
- 底部显示最多 3 条最近完成，均可点击恢复。
- “最近完成”作为明确的小节，使用简短标题和轻量分隔，不显示时间。

### 自适应与空间优先级

固定条数只是上限，实际布局按以下优先级使用空间：

1. 顶部操作栏
2. 至少一条待买物品
3. 规定数量的最近完成
4. 更多待买物品

物品名称的可读性高于标签：

- 名称保持单行并获得更高布局优先级。
- 空间不足时先截断标签名称，再减少仅颜色圆点的展示数量。
- 标签采用确定性顺序，即快照中的标签顺序；“第一个标签”不会因刷新随机变化。

这种布局不加入时间、无关统计或装饰卡片。内容不足时用紧凑分区和自适应行数减少“中间空”，但不伪造低价值信息。

## 3. 完成与最近完成恢复

### 完成

点击待买物品的圆圈：

1. 立即从本地共享快照中的待买列表移除。
2. 以当前时间加入最近完成列表顶部。
3. 将物品 ID 写入待处理完成队列。
4. 重载小组件 timeline。
5. 标记“需要尽快同步”并提交合并后的后台刷新请求。

### 恢复

点击最近完成物品：

1. 立即从本地共享快照中的最近完成列表移除。
2. 以原有排序信息放回待买列表。
3. 将物品 ID 写入待处理恢复队列。
4. 重载小组件 timeline。
5. 标记“需要尽快同步”并提交合并后的后台刷新请求。

主 App 获得执行机会后，批量消费完成与恢复队列，再发布数据库中的权威快照。重复 ID、已处于目标状态或已删除物品应安全忽略。

## 4. 快速添加

各尺寸顶部提供 `+`：

- 使用 `shop://add` 深链接打开主 App。
- iOS 与 macOS App 均处理该链接并直接呈现现有物品添加界面。
- 用户可输入名称、选择一个或多个已有标签，也可新建标签。
- 打开或回到前台时先触发现有同步流程。
- 用户保存后会触发现有本地 mutation observer，由 `SyncCoordinator` 经过约 2 秒防抖后发起一次同步；连续添加或修改会合并，不为每次字段变化重复联网。

若 App 已经打开，仍直接切换或弹出添加界面，不创建重复页面。

## 5. 每实例标签筛选

小组件从 `StaticConfiguration` 迁移到 `AppIntentConfiguration`。

编辑小组件时提供两个参数：

- 标签：支持多选。
  - 提供特殊选项“全部标签”。
  - 其余选项只列出当前至少关联一个待买物品的标签，减少无效配置项。
  - “全部标签”与具体标签同时被选中时，“全部标签”优先，等价于不筛选。
  - 未选择任何标签时也按“全部标签”处理，避免空配置得到空列表。
- 匹配方式：
  - 任一匹配（OR）
  - 全部匹配（AND）
  - 默认使用任一匹配。
  - 选择“全部标签”或未选择标签时，该参数不影响结果。

匹配规则同时应用于待买列表和最近完成列表：

- OR：物品拥有任意一个所选标签即显示。
- AND：物品必须拥有全部所选标签才显示。

WidgetKit 持久化每个实例的 configuration，因此主屏上的不同小组件可以长期保持不同标签组合与匹配方式，直到用户再次编辑该实例。

配置中的标签实体以稳定 UUID 为身份。标签改名后配置继续有效并展示新名称；标签删除后忽略失效 ID。若所有已选标签均失效，则回退到“全部标签”，避免小组件永久空白。

## 6. 后台同步策略

### 常规同步

- 07:00–22:00：每次后台任务完成后，请求下一次最早约 1 小时后执行。
- 22:00–07:00：请求下一次最早约 3 小时后执行。
- App 启动或回到前台：立即执行一次现有 `syncNowIfConfigured()`。
- WebDAV 未配置时，不发起联网请求，只刷新本地快照并安排下一次任务。

iOS 使用 `BGAppRefreshTask`；系统只把请求时间视为 `earliestBeginDate`，实际执行时间由电量、网络、使用习惯和系统预算决定。macOS 使用 `NSBackgroundActivityScheduler` 保持同样的分时策略与尽力而为语义；原生 Mac App 被用户完全退出后，后台活动不会像常驻服务一样继续运行，下一次启动或前台激活会立即补同步。

### 小组件操作后的合并刷新

连续完成或恢复多个物品时，不为每次点击创建独立后台任务：

- 共享存储记录 `needsSync = true` 和最近操作时间。
- 调度器使用固定任务标识提交一个后台刷新请求。
- 提交新请求前取消或替换同标识的旧请求，确保系统队列中最多有一个待执行刷新。
- 后续点击只更新队列和最近操作时间，不叠加多个任务。
- 后台任务启动后一次性消费所有 pending 操作，执行一次 WebDAV 同步，成功后清除 `needsSync`。
- 同步失败时保留 `needsSync`，交由下一次后台任务或前台启动重试。

“尽快刷新”仍受系统调度控制，不能保证小组件点击后立即联网。小组件本机 UI 的完成/恢复反馈不依赖后台任务，应立即发生。

### 调度续接

每次后台任务无论成功、失败或过期，都根据当前时段安排下一次常规刷新。应用进入后台时也确保存在一个待调度任务，避免任务链中断。

## 7. 本地化与可访问性

新增或调整以下本地化内容：

- 全部标签
- 任一匹配
- 全部匹配
- 最近完成
- 添加物品
- 恢复为待买
- 当前筛选结果数量

交互要求：

- `+` 提供“添加物品”可访问性标签。
- 完成按钮读出物品名称与“标记为已完成”。
- 最近完成按钮读出物品名称与“恢复为待买”。
- 标签颜色圆点不能作为唯一语义来源；完整标签名称仍通过 accessibility value 暴露。
- 支持动态字体可用空间变化，标签优先收缩而不是物品名称。

## 8. 错误与边界情况

- App Group 不可用或快照损坏：显示空状态，不崩溃。
- 没有待买但有最近完成：仍显示最近完成区和 `+`。
- 筛选后没有待买但有匹配的最近完成：显示筛选空状态并保留最近完成区。
- 无匹配内容：显示简短筛选空状态，保留 `+`。
- 标签过多：仅显示规定数量的名称与可容纳的颜色圆点。
- 小组件操作后物品被其他设备删除：主 App 消费队列时安全忽略，下一次权威快照纠正本地乐观状态。
- WebDAV 同步失败：不回滚小组件本地操作；保留待处理状态并重试。

## 9. 测试计划

### 共享层单元测试

- 新旧快照格式兼容解码。
- 最近完成按 `completedAt` 倒序且数量受限。
- 小尺寸标签转换为“一个名称 + 其余色点”。
- 中、大尺寸标签转换为“最多三个名称 + 其余色点”。
- OR、AND、“全部标签”、空选择和失效标签筛选。
- 完成与恢复 pending 队列去重。
- 连续多次操作仅产生一个待同步调度状态。
- 同步成功清除 `needsSync`，失败保留。
- 07:00、22:00 及跨午夜边界选择正确的下一次间隔。

### 小组件测试

- 三种尺寸在浅色、深色和不同动态字体下无截断溢出。
- 2×2 不显示 App 名称。
- 不同小组件实例保存不同标签组合及 OR/AND 条件。
- 标签改名、删除后配置行为符合回退规则。
- 完成后待买条目立即消失并进入最近完成。
- 恢复后最近完成立即消失并回到待买。
- `+` 在 iOS 与 macOS 都打开添加页。
- 无待买、无最近完成、仅最近完成、筛选无结果等空状态。

### 后台与集成测试

- 前台启动立即同步并发布快照。
- 常规后台任务执行后续排下一次任务。
- 白天与夜间采用不同 `earliestBeginDate`。
- 快速连续勾选三项只保留一个后台任务请求，并一次消费三项。
- 后台任务过期时正确结束并保留重试状态。
- WebDAV 未配置、离线、认证失败时不丢 pending 操作。

## 10. 已确认决策

- 2×2 隐藏“Shop!”名称。
- 2×2 显示第一个标签名称，其余标签仅显示颜色。
- 2×4 与 4×4 最多显示三个标签名称，其余标签仅显示颜色。
- 2×2、2×4 分别显示一条最近完成；4×4 显示三条。
- 最近完成可直接恢复为待买。
- `+` 打开 App 添加页，可输入名称并选择或新建标签。
- 不显示时间，不增加小组件刷新按钮。
- 标签筛选通过“编辑小组件”配置，每个实例独立持久化。
- 标签支持多选，并由用户选择 OR 或 AND。
- 平时白天约每小时、夜间约每三小时请求后台刷新。
- 打开或回到前台立即同步。
- 小组件连续操作合并为一次尽快后台刷新请求。
